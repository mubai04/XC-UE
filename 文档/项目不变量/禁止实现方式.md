# 禁止实现方式

## 1. 文件职责

本文只保存稳定禁止机制、判定条件、检测方法和失败状态。本文不记录当前代码行号，不兼任专项审计报告，不把待审计风险直接判定为缺陷。

## 2. 统一治理裁决语义

本节适用于 F-01 至 F-13 的所有禁止实现方式。

已确认实现违规：证据足以证明实现实际违反某项禁止规则时，治理裁决为：

```text
REJECTED_IMPLEMENTATION
实现已驳回
```

不得使用 `NOT_PROVEN` 或运行时错误状态代替已确认违规的治理裁决。

计划本身不足：当前计划没有定义正确机制、唯一真源、允许固定范围、错误处理、正式执行链、迁移边界、修复边界或验收边界，因此无法安全继续实现时，治理裁决为：

```text
BLOCKED_FOR_PLAN_REVISION
因计划需要修订而阻断
```

该状态表示计划需要修订，不表示当前实现已经通过。

证据不足：现有证据不足以证明是否违规，调查没有覆盖实际执行链，或缺少输入、调用路径、差异、哈希、日志、可复现证据时，治理裁决为：

```text
NOT_PROVEN
未证明
```

`NOT_PROVEN` 只表示无法下结论，不得用于已经确认的违规。

运行时状态：`FAILED_SCHEMA`、`FAILED_INTERNAL`、业务拒绝状态、对应结构化错误状态、退出码和异常信封属于运行结果或错误证据，不得直接替代治理裁决。必须先根据证据判断是否确认违反禁止规则、是否只是计划不足、是否只是证据不足，再使用对应治理裁决。

优先级：同一问题同时满足多个条件时，已确认实现违规优先使用 `REJECTED_IMPLEMENTATION`；尚未确认违规但计划无法支持正确实施，使用 `BLOCKED_FOR_PLAN_REVISION`；既未确认违规，也无法确认计划问题，且证据不足，使用 `NOT_PROVEN`。不得选择较弱状态规避较强裁决。

## F-01 写死输出

定义：实现中直接返回固定结论、固定报告、固定哈希、固定状态或固定业务结果，而不是由输入、规则和执行链计算得出。

禁止原因：写死输出会伪装动态能力，使测试和验收无法证明真实机制。

判定条件：更换有效输入后，输出仍保持与输入无关的固定结论；或实现绕过规则加载、Schema 校验和主执行链直接构造验收结果。

检测方法：使用不同输入、不同规则或不同项目上下文运行同一入口，核对输出是否随输入和规则变化；检查实现是否存在绕过主链路的固定返回。

失败状态：

- 已确认存在写死输出：`REJECTED_IMPLEMENTATION`。
- 计划未定义输出必须如何由输入、规则和执行链产生：`BLOCKED_FOR_PLAN_REVISION`。
- 证据不足以判断输出是否写死：`NOT_PROVEN`。

## F-02 针对样例特判

定义：实现针对测试样例、固定文件名、固定片段、固定 run id 或固定路径编写特殊分支。

禁止原因：样例特判无法推广到真实输入，会让测试通过但机制不成立。

判定条件：更换等价样例或路径后行为失效；或分支条件只识别测试样例而不表达通用规则。

检测方法：替换同构输入、随机化运行编号、改变非语义路径，核对行为是否仍由规则驱动。

失败状态：

- 已确认存在样例、固定文件、固定片段、固定运行编号或固定路径特判：`REJECTED_IMPLEMENTATION`。
- 计划未定义通用规则、合法固定范围或泛化要求：`BLOCKED_FOR_PLAN_REVISION`。
- 证据不足以判断是否存在特判：`NOT_PROVEN`。

## F-03 建立平行真源

定义：在既有运行规则、Schema、协议或治理真源之外，新增另一套可驱动实现或验收的事实来源。

禁止原因：平行真源会造成冲突，破坏治理边界和外部验收依据。

判定条件：同一事实在多个文件中以可执行或验收依据形式重复维护；或 README、INDEX、AGENTS 保存本应由项目架构、项目状态或规则文件唯一维护的动态事实。

检测方法：检查治理文件职责边界；检查实现是否从未授权清单、旧报告、缓存、归档或运行产物读取当前规则。

失败状态：

- 已确认建立平行真源或从未授权来源驱动实现、状态或验收：`REJECTED_IMPLEMENTATION`。
- 计划没有明确唯一真源、迁移关系或旧真源退役方式：`BLOCKED_FOR_PLAN_REVISION`。
- 证据不足以判断是否存在平行真源：`NOT_PROVEN`。

## F-04 吞掉异常

定义：捕获异常后不报告、不封装、不传播，直接返回空值、默认成功或不可区分的普通结果。

禁止原因：异常被吞掉会掩盖规则缺失、解析失败、路径越界和证据不足。

判定条件：异常路径无法在退出码、错误信封、日志或报告中定位；或失败输入被转换为成功状态。

检测方法：构造缺失文件、非法 JSON、非法路径、Schema 错误和编码错误，核对是否产生结构化失败证据。

失败状态：

- 已确认异常被吞掉、失败被转换为成功，或错误无法被定位：`REJECTED_IMPLEMENTATION`。
- 计划没有定义异常传播、错误信封、退出码或失败路由：`BLOCKED_FOR_PLAN_REVISION`。
- 证据不足以判断是否吞掉异常：`NOT_PROVEN`。
- `FAILED_SCHEMA`、`FAILED_INTERNAL` 及其他结构化错误状态仅作为运行时证据，不替代治理裁决。

## F-05 绕过主执行链

定义：不经过统一入口、项目加载、生产资格、规则加载、Schema 校验或授权执行器，直接读写关键产物或正文。

禁止原因：绕过主链路会使权限、血缘、回滚和验收证据失效。

判定条件：关键产物不是由声明的入口生成；或产物缺少输入快照、规则血缘、Schema 校验和运行清单。

检测方法：核对产物路径、生产者、运行编号、输入哈希、规则哈希和 Schema 校验记录。

失败状态：

- 已确认绕过统一入口、项目加载、资格校验、规则加载、Schema 校验或授权执行器：`REJECTED_IMPLEMENTATION`。
- 计划没有明确规定正式执行链或必须经过的闸门：`BLOCKED_FOR_PLAN_REVISION`。
- 证据不足以判断是否绕过主执行链：`NOT_PROVEN`。

## F-06 修改测试迁就实现

定义：为了让当前实现通过而降低、删除或改写测试要求，而不是修复机制或修订已批准计划。

禁止原因：测试迁就实现会破坏回归证据和外部验收可信度。

判定条件：测试断言被放宽但计划、规则或需求没有同步批准；或失败行为未修复而测试改为接受失败。

检测方法：审查测试变更与批准计划、规则真源、缺陷证据之间的一致性。

失败状态：

- 已确认通过降低、删除或改写测试要求迁就错误实现：`REJECTED_IMPLEMENTATION`。
- 当前需求、测试和计划互相冲突，需要重新确定验收标准：`BLOCKED_FOR_PLAN_REVISION`。
- 证据不足以判断测试变更是否在迁就实现：`NOT_PROVEN`。

## F-07 静态产物伪装动态能力

定义：用预先生成的报告、缓存、旧运行输出或手写产物冒充当前执行结果。

禁止原因：静态产物不能证明当前代码、规则和输入链路仍可复现。

判定条件：产物时间、运行编号、输入哈希、规则哈希或生成入口与当前任务不一致；或无法通过命令重新生成。

检测方法：重新执行指定命令，比较产物血缘、哈希、运行编号和工作树变化。

失败状态：

- 已确认使用旧报告、缓存、预制文件或手写产物冒充当前动态执行结果：`REJECTED_IMPLEMENTATION`。
- 计划没有定义动态执行、产物血缘、可复现性或当前证据要求：`BLOCKED_FOR_PLAN_REVISION`。
- 证据不足以判断产物是否由当前执行产生：`NOT_PROVEN`。

## F-08 未经批准修改正式正文

定义：在没有明确授权、冻结补丁和复验条件的情况下，手工编辑、改写或替换小说正式正文。

禁止原因：正式正文是受控内容，未经批准修改会破坏项目事实和回滚边界。

判定条件：正式正文文件在未授权任务中发生内容变化；或运行时写入缺少审批、哈希匹配、复验和回滚证据。

检测方法：检查 Git diff、运行审计、审批记录、哈希字段和复验结果。

失败状态：

- 已确认在未获授权、补丁未冻结或复验条件不满足时修改正式正文：`REJECTED_IMPLEMENTATION`。
- 计划没有明确正文权限、批准条件、哈希要求、复验条件或回滚边界：`BLOCKED_FOR_PLAN_REVISION`。
- 证据不足以判断正文修改是否获得批准：`NOT_PROVEN`。
- 运行时正文拒绝状态只作为执行证据，不替代治理裁决。

## F-09 旧证据冒充当前证据

定义：使用归档、缓存、旧报告、旧测试输出或旧输入作为当前验证结论。

禁止原因：旧证据不能证明当前代码、规则、输入和环境。

判定条件：报告中引用的证据不是本轮命令产生，或无法复现。

检测方法：核对命令、时间、路径、运行编号和工作树状态。

失败状态：

- 已确认使用归档、缓存、旧报告、旧测试输出或旧输入冒充当前证据：`REJECTED_IMPLEMENTATION`。
- 计划没有定义证据时间边界、运行编号、输入版本或当前证据要求：`BLOCKED_FOR_PLAN_REVISION`。
- 证据不足以判断引用证据是否属于当前任务：`NOT_PROVEN`。

## F-10 固定项目或固定路径

定义：在生产实现中写死项目 ID、项目目录、目标文件、章节路径、测试项目名称，或把只适用于特定受控测试项目的实现宣称为通用能力。

禁止原因：固定项目或固定路径会把受限范围伪装成通用机制，使多项目、可配置或可迁移能力无法成立。

判定条件：允许情况是固定范围已经在冻结计划、项目注册表、结构化配置或受控测试范围中明确声明，并且实现没有把该固定范围伪装成通用能力。禁止情况是任务要求通用、可配置或多项目能力，但实现直接内置固定项目 ID、固定章节名、固定绝对或相对路径、固定项目目录、固定测试数据，或针对一个已知文件返回预设结果。若固定范围来自计划本身不清楚，不能继续补丁实现。

检测方法：搜索固定项目 ID 和固定路径常量；更换项目 ID 后复验；更换目标文件或章节后复验；检查路径是否来自项目注册表、参数或结构化配置；检查实现是否把受限能力报告为通用能力。

失败状态：`REJECTED_IMPLEMENTATION`。若固定范围来自计划本身不清楚，返回 `BLOCKED_FOR_PLAN_REVISION`。

## F-11 测试环境特判

定义：生产代码根据是否由 pytest 或测试框架调用、测试文件名、测试函数名、调用栈中出现测试代码、特定测试环境变量、只在测试中使用的固定输入，或已知断言，改变真实业务逻辑并返回预设结果。

禁止原因：测试环境特判会让测试路径不同于真实运行路径，使测试通过无法证明生产机制成立。

判定条件：测试目录中的 fixture、mock 和测试辅助代码不属于禁止对象。禁止对象是生产实现为了让测试通过而识别测试环境，并走不同于真实运行的业务路径；或生产实现只在测试信号存在时跳过真实规则、权限、数据流、错误处理或执行链。

检测方法：搜索 pytest、unittest、测试文件名和测试环境变量；检查生产代码中的调用栈判断；在非测试入口复现同一输入；比较测试环境和生产环境的执行链；使用未知样例和变形输入复验。

失败状态：`REJECTED_IMPLEMENTATION`。

## F-12 从归档或旧输入反向覆盖活跃工程

定义：从归档区读取旧代码、旧规则、旧报告或旧测试产物，并用历史文件覆盖当前活跃代码、结构化规则、Schema、项目状态或验收证据；或未经迁移程序和批准，将旧输入重新指定为当前真源；或根据旧报告回退已经确认的当前事实。

禁止原因：归档、整改基线、旧输入和运行记录可以用于历史研究，但不能因此获得当前真源资格；反向覆盖会破坏当前事实、证据链和回滚边界。

判定条件：允许读取归档进行历史研究。只有在有明确迁移任务、来源与目标确定、冲突处理规则确定、当前内容已备份或可回滚、有人工批准、有迁移后验证证据、真源资格重新登记全部满足时，才允许正式迁移。若确实需要恢复旧内容但没有迁移计划，不能继续实施。

检测方法：检查活跃文件的来源；检查是否从归档、整改基线、旧输入或运行记录复制内容；对比修改时间、哈希和 Git 差异；检查是否存在明确迁移计划；检查旧文件是否被重新加入默认索引或运行链。

失败状态：`REJECTED_IMPLEMENTATION`。若确实需要恢复旧内容但没有迁移计划，返回 `BLOCKED_FOR_PLAN_REVISION`。

## F-13 只修表现而不处理根因

定义：只修改最终输出而不修复产生错误的机制；只增加条件分支绕过失败；只屏蔽错误提示；只吞掉异常；只针对当前样例补丁；复制、重复或填充内容以达到阈值；原文复制后宣称已经生成或修复；或在错误执行链外增加旁路，制造表面正确结果。

禁止原因：只修表现会保留根因，使同类问题在未知输入、真实执行链或后续阶段继续复现。

判定条件：原始故障仍能通过其他输入复现；修复没有作用于根因所在层；正式执行链仍然错误；仅已知样例通过；删除补丁后根因完全未改变；完成报告只有输出结果，没有执行机制证据。临时止血只有在明确标记为临时措施、不宣称根因已解决、登记剩余缺陷和技术债、规定移除条件、不绕过安全权限和数据一致性闸门、获得明确批准时才允许。若现有计划没有识别根因或指定正确修复层，不能继续实施。

检测方法：重新复现原始故障；使用不同输入和未知样例；检查真实执行链；对比修复前后根因层状态；检查是否新增旁路、重复文本、固定返回值或吞错；检查完成报告是否提供根因修复证据。

失败状态：`REJECTED_IMPLEMENTATION`。若现有计划没有识别根因或指定正确修复层，返回 `BLOCKED_FOR_PLAN_REVISION`。
