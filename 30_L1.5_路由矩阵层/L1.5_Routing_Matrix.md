## 1. 定位

L1.5 是 L1 失败后的路由矩阵。

它只负责一件事：

把 L1 输出的失败类型，转译成明确的修复作业、建议修复方向、输出产物和回流验收位置。

L1.5 不直接判断作品是否成立。
L1.5 不直接判断读者是否投入。
L1.5 不直接判断章节是否可以发布。
L1.5 不直接写正文。
L1.5 不替代 L1-01 / L1-02 / L1-03。
L1.5 不替代 L2 能力模块。
L1.5 不替代 L3 Cursor 执行协议。
L1.5 不替代人工判断。

---

## 2. 核心问题

L1.5 回答一个问题：

```text
这个失败，应该派到哪里修，修完回哪里验？
```

L1 判断失败。
L1.5 分配修复路径。
L2、人工或技术修正执行。
修复后回原 L1 闸门复验。

---

## 3. L1.5 的判断边界

L1.5 不做作品质量判断。

L1.5 只做路由判断。

路由判断包括：

```text
从 L1 输出的失败中确定唯一主失败类型；
确定该失败对应的修复作业；
确定建议修复方向；
确定修复产物；
确定回流验收位置。
```

L1.5 不得重新评判：

```text
作品内部是否成立；
读者是否愿意继续；
章节是否可以发布；
正文写得好不好；
项目是否值得继续。
```

这些判断必须回到对应 L1 闸门或人工复核。


## 7. 路由总原则

### 7.1 原闸门回流原则

默认规则：

从哪个 L1 闸门失败，就回哪个 L1 闸门复验。

```text
L1-01 失败 → 修复后回 L1-01
L1-02 失败 → 修复后回 L1-02
L1-03 失败 → 修复后回 L1-03
```

但该原则受“前置闸门失败优先原则”覆盖。

---

### 7.2 前置闸门失败优先原则

如果后置闸门失败的根因来自前置闸门未通过，则必须先回前置闸门。

```text
L1-02 失败原因是 L1-01 未通过
→ 主回流验收位置为 L1-01
→ L1-01 通过后再返回 L1-02

L1-03 失败原因是 L1-02 未通过 / 投入意愿不足
→ 主回流验收位置为 L1-02
→ L1-02 通过后再返回 L1-03

L1-03 失败原因是 L1-01 内部结构未成立
→ 主回流验收位置为 L1-01
→ L1-01 通过后再走 L1-02，再回 L1-03
```

禁止在前置闸门未通过时，强行给后置闸门判定通过。

---

### 7.3 待建能力标注原则

L1.5 不直接执行 L2。

L1.5 只写建议修复方向：

```text
建议修复方向：待建 L2 文风系统
建议修复方向：待建 L2 叙事系统
建议修复方向：待建 L2 角色系统
建议修复方向：待建 L2 创意设定系统
建议修复方向：待建 L2 AI味控制系统
建议修复方向：待建 L2 章节生成系统
建议修复方向：待建 L2 读者价值系统
建议修复方向：技术修正 / 待建 L3 文件结构协议
建议修复方向：人工复核
```

在对应 L2 文件未建前，所有 L2 方向只能作为“待建能力方向”，不得标记为已调用。

---

### 7.4 唯一路径原则

每次 L1.5 路由必须确定一个主失败类型。

允许记录次级失败类型，但只能有一个主处理路径。

如果多个失败同时存在，按以下优先级决定主失败类型：

```text
输入不足
↓
技术护栏失败
↓
L1-01 未通过
↓
功能锁失败
↓
发布锁失败
↓
读者投入失败
↓
文风 / 叙事 / 角色 / 创意设定 / AI味专项失败
```

输出中必须明确：

```text
主失败类型：
次级失败类型：
```

禁止同时输出多个主失败类型。

---

### 7.5 阻断优先原则

如果前置闸门未通过，后置闸门不得强行通过。

```text
L1-01 未通过 → L1-02 不得直接通过
L1-02 未通过 → L1-03 不得直接发布
L1-03 未通过 → 不得进入发布执行
```

---

### 7.6 复合失败仲裁原则

真实章节常会同时出现多个失败。

L1.5 必须从复合失败中压出一个主失败类型。

#### 7.6.1 E / V / C 复合失败仲裁

```text
如果 C高 已经压过 E 和 V，主失败类型为 C高。
如果 C 不高，但读者不知道下一章看什么，主失败类型为 V低。
如果 V 成立但当章没有即时反应，主失败类型为 E低。
如果入口弱导致 E / V 都低，主失败类型为入口弱。
如果章末弱导致 V低，主失败类型为章末弱。
```

#### 7.6.2 内部创作复合失败仲裁

```text
如果角色动机不成立导致叙事无法推进，主失败类型为角色失败。
如果设定没有压力导致冲突无法成立，主失败类型为创意设定失败。
如果因果链断裂导致爽点错位，主失败类型为叙事失败。
如果语言问题已经影响信息理解，主失败类型为文风失败。
如果 AI味主要表现为解释腔、同义反复、总结腔，主失败类型为 AI味失败。
```

#### 7.6.3 发布锁复合失败仲裁

```text
如果功能锁失败，优先功能锁失败。
如果功能锁通过但当章收益不足，主失败类型为当章收益不足。
如果当章收益存在但章末没有追读，主失败类型为章末追读弱。
如果字数不足但已有有效收益，主失败类型为字数不足。
如果字数不足且事件本身无收益，主失败类型为当章收益不足。
如果认知成本过高导致发布风险，主失败类型为认知成本过高。
```

---

### 7.7 人工复核原则

以下情况必须进入人工复核：

```text
失败类型互相冲突
同一问题连续两轮修复失败
L1.5 无法确定唯一主路径
正文存在明显价值但闸门无法判定
需要删项目 / 冻结项目 / 换项目
L1.5 判断与原 L1 输出冲突
```

人工复核不等于直接放行。

人工复核后仍需回到对应 L1 闸门复验。

## 10. L1-03 路由矩阵

L1-03 负责发布锁验收。

L1-03 的失败类型由发布判定项映射而来。
L1.5 不在此处重复维护 L1-03 的完整失败枚举。

| L1-03 失败类型 / 发布判定项 | 作业场景            | 建议修复方向                     | 修复产物       | 回流验收位置 |

## 17. 验收条件

通过以下测试前，本文件不得定版：

```text
_tests/L1-L1.5_当前系统十个失败场景路由测试.md
```

测试要求：

```text
10 个失败场景全部能进入 L1.5
至少 8 个失败场景能输出唯一主失败类型
至少 8 个失败场景能输出唯一回流验收位置
不得出现待建文件被当作已启用文件
不得出现 L1.5 直接写正文
不得出现 L1.5 绕过原 L1 复验
不得出现多个主失败类型并列
不得出现前置闸门失败却强行回后置闸门
```

## 18. 最小测试场景

第一轮至少测试以下 10 个场景：

```text

2. 技术护栏失败
3. 文风失败
4. 叙事失败
5. 角色失败
```

## 20. 一句话总纲

L1.5 不做作品质量判断，不写文，不发布。
