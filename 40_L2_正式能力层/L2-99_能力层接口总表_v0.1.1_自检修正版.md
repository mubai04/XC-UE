# L2-99 能力层接口总表 v0.1.1

> 状态：候选真源 / 未定版 / 待审计  
> 层级：L2 能力层接口总表  
> 文件性质：接口总表，不是新增能力模块。  
> 真源规则：Markdown 为真源；图片、提示词、对话结论不得覆盖本文件。  
> 版本说明：v0.1.1 为 v0.1 自检修正版，重点修正“接口建议误写成派单裁决”“混合问题处理不足”“L2 / L3 / 外部运营边界不够硬”等问题。

---

## 0. 自检修正摘要

v0.1 存在以下隐患：

1. “建议能力模块”容易被误解为 L2-99 替 L1.5 派单。  
   修正：统一改为“接口候选模块”。L2-99 只给候选归属，不拥有最终路由裁决权。

2. “统一输入接口”没有区分直接来自 L1 的失败包与 L1.5 已裁决失败包。  
   修正：新增“输入来源模式”：直接闸门输入 / L1.5 已路由输入 / 混合问题回流输入。

3. “跨模块冲突规则”仍偏说明型，缺少执行顺序。  
   修正：增加“冲突仲裁顺序”：先 L1.5、再主失败、再次失败、再接口回流。

4. L2-99 可能被误用成总控系统。  
   修正：增加“接口层四不原则”：不裁决、不修复、不生成、不覆盖。

5. 对 L3 和外部运营层的边界不够强。  
   修正：单独定义“非 L2 出口”。

6. 缺少“接口失败类型”。  
   修正：新增 IF-P1 至 IF-P10，专门用于识别接口层错误，但不作为创作能力失败类型。

7. 缺少“接口验收状态”。  
   修正：新增接口判断最终状态六类。

## 1. 文件定位

L2-99 不是新的创作能力。

**输入边界真源**：见 `L2_输入与证据上下文边界.md`（L2-BND-V2）。

L2-99 是 L2 能力层的接口总表，负责回答：

1. L1 / L1.5 派来的问题，可能进入哪个 L2。
2. 每个 L2 接收什么输入。
3. 每个 L2 输出什么修复产物。
4. 修完回哪里验收。
5. 哪些问题必须回 L1.5 重路由。
6. 哪些问题不能被 L2 处理，必须进入 L3 或外部运营层。
7. 多个 L2 同时相关时，如何避免乱修、错修、越界修。

一句话：

> L2-99 = L2 能力层的调用索引、边界索引、回流索引。

---

## 2. 接口层四不原则

L2-99 必须遵守四不原则：

1. 不裁决：不替 L1.5 决定最终主失败归属。  
2. 不修复：不直接执行任何 L2 修复动作。  
3. 不生成：不生成正文、章节、大纲、设定、角色、市场方案。  
4. 不覆盖：不允许图片、对话、旧文件覆盖 Markdown 真源。

L2-99 只能输出：

- 接口候选模块
- 边界判断
- 回流建议
- 输入 / 输出格式
- 冲突处理规则
- 非 L2 出口判断

## 6. 输入来源模式

L2-99 必须先判断输入来源模式。

### 6.1 直接闸门输入

来源：

- L1-00
- L1-01
- L1-02
- L1-03

处理原则：

- 只做接口候选判断。
- 如果主失败不清，必须回 L1.5。
- 不得直接强派 L2。

### 6.2 L1.5 已路由输入

来源：

- L1.5 Routing Matrix

处理原则：

- 尊重 L1.5 主失败裁决。
- L2-99 只校验接口是否匹配。
- 若发现明显越界，只能建议回 L1.5 复判。

### 6.3 混合问题回流输入

来源：

- L2 修复中发现其他能力层问题
- 图文一致性审计
- 测试断点
- 失败复盘

处理原则：

- 不做跨模块修复。
- 标注主模块 / 次模块候选。
- 建议回 L1.5 重新拆包。

## 7. L2 调用总原则

### 7.1 单一主失败原则

能归一个 L2，就不要跨 L2。

跨 L2 问题必须拆成：

- 主失败
- 次失败
- 回流验收位置

### 7.2 L1.5 裁决权原则

主失败由 L1.5 决定。

L2-99 只提供接口规则，不替 L1.5 裁决。

### 7.3 能力层边界原则

L2 只修能力问题，不写正文。

L2 发现越界，只能建议回 L1.5 重路由。

### 7.4 真源优先原则

Markdown 真源高于图片。

图片只做架构表达，不得覆盖文件定义。

### 7.5 非 L2 出口原则

属于以下问题时，不进入 L2：

- Cursor 执行
- 脚本
- 文件落盘
- 目录操作
- 调用协议
- 封面
- 书名
- 简介
- 投流
- 平台推荐
- 真实收入
- 真实签约率

前五类进入 L3。  
后七类进入外部运营层。

## 11. 跨模块冲突规则

### 11.1 L2-01 vs L2-06

L2-01 负责结构如何剪枝。  
L2-06 负责记录哪些可能性已经开放、关闭、暂存、禁止回潮。

判断：

- “应该删哪条路” → L2-01
- “哪条路已经被删，后文不能回潮” → L2-06

### 11.2 L2-02 vs L2-05

L2-02 负责句级语言是否值得读。  
L2-05 负责指定时间窗口内体验收益是否足够。

判断：

- “这句话是不是 AI 味、重复、无增量” → L2-02
- “这一段是否让读者觉得继续读有收益” → L2-05

### 11.3 L2-03 vs L2-01

L2-01 负责“他要做什么”在结构上是否成立。  
L2-03 负责“他为什么非要这样做”在心理上是否成立。

判断：

- “章节目标轴不清” → L2-01
- “人物非要这么做不可信” → L2-03

### 11.4 L2-04 vs L2-06

L2-04 负责规则、中心价值和设定玩法如何成立。  
L2-06 负责这些规则、价值和后果是否被继承、约束和回写。

判断：

- “规则本身不成立” → L2-04
- “规则前后不继承、状态漂移” → L2-06

### 11.5 L2-05 vs 外部运营层

L2-05 负责内容侧用户体验收益。  
外部运营层负责封面、书名、渠道、投流、平台推荐、真实收入。

判断：

- “读者读正文时没有继续收益” → L2-05
- “封面点击率、书名吸引力、平台推荐” → 外部运营层

## 12. 冲突仲裁顺序

当一个问题同时命中多个模块时，按以下顺序处理：

1. 先判断是否属于 L3 或外部运营层。  
2. 再判断是否需要回 L1.5。  
3. 再判断是否已有 L1.5 主失败裁决。  
4. 若已有裁决，尊重主失败模块。  
5. 若没有裁决，只输出接口候选模块。  
6. 若两个模块都强相关，标主候选与次候选，回 L1.5。  
7. 若问题本身输入不足，输出“需补输入”。  
8. 若图片与 Markdown 冲突，回 L2-06 真源校验。

## 14. 接口失败类型 IF-P1 至 IF-P10

接口失败类型不是创作失败类型，只用于 L2-99 自检。

| 编号 | 接口失败类型 | 处理 |
|---|---|---|
| IF-P1 | 输入来源不明 | 补输入 |
| IF-P2 | 主失败不清 | 回 L1.5 |
| IF-P3 | 模块候选冲突 | 回 L1.5 |
| IF-P4 | L2 / L3 混淆 | 判非 L2 出口 |
| IF-P5 | L2 / 外部运营层混淆 | 判非 L2 出口 |
| IF-P6 | L2-06 被误写成总控 | 回 L2-06 边界复核 |
| IF-P7 | L2-05 被误写成市场运营 | 回 L2-05 边界复核 |
| IF-P8 | 图片覆盖 Markdown | 回 L2-06 真源校验 |
| IF-P9 | 对话临时结论覆盖文件 | 回 L2-06 真源校验 |
| IF-P10 | 新增未定义能力模块 | 阻断，回 L2-00 复核 |

## 15. 接口判断最终状态

L2-99 输出必须落入以下状态之一：

1. 接口明确，可进入对应 L2。
2. 主失败不清，回 L1.5。
3. 输入不足，需补输入。
4. 非 L2 问题，进入 L3。
5. 非 L2 问题，进入外部运营层。
6. 真源冲突，回 L2-06。
7. 需人工复核。
8. 阻断，不得进入 L2。
