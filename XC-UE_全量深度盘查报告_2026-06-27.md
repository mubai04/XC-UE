# 三、设计层问题

## P0-01：项目身份设计断裂——“已注册”但“不可加载”

**证据：**

- `项目注册表.json` 的 `default_project` 是 `TP-001`；
- TP-001 注册项指向 `70_测试项目/TP-001_CleanHarness_IR_Runtime`；
- `项目加载器.py` 固定读取 `<project_root>/project.json`；
- 实际目录没有任何 `project.json`；
- 统一入口运行 TP-001 返回 exit 27。

**问题本质：** 项目注册表、Project Harness 文档结构和可执行项目清单并未真正接合。

**影响：**

- 默认项目不可运行；
- L1 语义上下文无法获得 manifest 级章节顺序；
- `PROJECT` 目标也无法用于该 Harness；
- 系统所谓“项目化”当前只在加载器代码中成立，在测试项目资产中不成立。

**整改：**

二选一，不得继续保持矛盾状态：

1. 为 TP-001 增加符合 `xcue.project-manifest/1.0` 的 `project.json`，并补齐 entrypoint；或
2. 在 TP-001 未完成前，从注册表移除 default_project，禁止把空壳注册成可运行项目。

**验收：** 使用 `--project TP-001` 可完成项目解析，且项目 ID、正文根、默认章节、章节顺序和入口均通过路径边界校验。

**置信度：100%。**

## P0-02：产品形态没有做出唯一决定

当前同时存在三种形态：

1. `src/xc_ue`：现代 Python 包壳；
2. `00_工程总控/工程执行层`：实际引擎；
3. 根目录 Markdown 层：被声明为系统架构真源。

`src/xc_ue` 的 CLI 与实际统一运行入口无直接连接；实际引擎靠大量 `sys.path.insert()` 导入中文目录模块。最终 wheel 不包含引擎。

**设计问题：** 团队尚未决定 XC-UE 是：

- 一个只能在仓库根目录运行的本地工作区；还是
- 一个可安装、可调用、可版本化的 Python 产品。

现在的实现同时暗示二者，却都不完整。

**建议：**

优先选择“可安装本地应用”路线：

```text
src/xc_ue/
├─ cli.py
├─ l1/
├─ l2/
├─ l3/
├─ common/
├─ schemas/
└─ resources/
```

Markdown 标准继续留在 `docs/`，但运行代码不再依靠任意 `sys.path` 注入。旧中文文件名可以保留在文档层；Python 包内部建议使用稳定模块名或建立清晰适配层。

若坚持“仓库工作区应用”，则应删除误导性的可发布包入口和 wheel 配置，明确只支持仓库根运行。

**置信度：99%。**

## P0-03：系统没有统一的“真源优先级矩阵”

根 README 声明：

```text
系统层 10_L0~50_L3 = XC-UE 架构真源（Markdown）
Markdown > 图片
```

但实际运行时：

- L1 读取 `gate_rules.json`；
- L2 读取 `ability_rules.json`、`routes.json`；
- L3 读取 `protocol_rules.json`；
- 代码行为才决定真实分支；
- 状态又写在多个 README 和 CURRENT_SYSTEM_STATUS 中。

因此至少存在四类“真源”：

1. 架构意图：Markdown；
2. 可执行规则：JSON；
3. 实际行为：Python；
4. 运行事实：报告/归档。

当前没有一份机器可校验的优先级表说明它们冲突时谁胜出。

**建议真源层级：**

```text
运行事实（报告 + 哈希）
  > 可执行规则（JSON Schema 验证）
  > 代码实现
  > 架构规范（Markdown）
  > 解释性说明与历史文档
```

“代码实现”和“规则 JSON”谁优先需按模块进一步明确：JSON 决策数据应高于代码内置默认；代码负责解释规则，不能暗藏第二套路由。

**置信度：98%。**

## P1-01：状态枚举混乱

仓库中同时使用：

- `active`
- `candidate`
- `CANDIDATE_TEST`
- `production_eligible`
- `PILOT_VALIDATED`
- `MODEL_CALIBRATION_REJECTED`
- `IMPLEMENTED`
- `冻结`
- `候选真源`
- `正式能力`

这些词属于不同层级，却没有统一状态机。例如规则文件 `status=active`，但没有 `production_eligible=true`；目录名叫“正式能力层”，README 又称“候选真源，未定版”。

**建议：** 建立一套正交状态：

```text
implementation_status: NOT_STARTED | PARTIAL | IMPLEMENTED
validation_status: UNTESTED | UNIT_VALIDATED | PILOT_REJECTED | PILOT_VALIDATED
release_status: EXPERIMENTAL | CANDIDATE | PRODUCTION
freeze_status: MUTABLE | FROZEN
```

不得再用一个“已完成”覆盖实施、验证、发布和冻结四种不同含义。

**置信度：96%。**

# 四、结构与代码组织问题

## P0-04：wheel 是空壳，安装结果与仓库运行结果不同

`pyproject.toml` 只发现 `src/xc_ue*`。真实引擎不在包目录，规则和 Schema 也没有配置为 package data。

**后果：**

- `pip install` 成功会制造“安装成功”的假象；
- `xcue` 命令只能执行壳 CLI，无法代表仓库引擎；
- 依赖仓库相对路径的逻辑无法在已安装环境中复现；
- 发布版本 `0.0.0` 没有任何真实可追溯性。

**验收标准：** 新建干净虚拟环境，只安装 wheel，不复制仓库，仍能运行 `xcue l1 ...`、加载 Schema/规则，并通过最小集成测试。

## P1-02：大量 `sys.path` 注入是架构债务

统一入口、L1、L2、L3、评估脚本等均通过运行时修改 `sys.path` 找模块。这会造成：

- 导入依赖当前文件位置；
- 同名模块覆盖风险；
- IDE、类型检查、测试与安装环境行为不一致；
- 难以确定模块公共 API；
- 外部插件或项目入口更难安全隔离。

**建议：** 迁移前先建立模块依赖图；禁止边迁移边重写业务逻辑。以公共组件为第一批包化对象，再迁 L1、L2、L3。

## P1-03：生成物和本地环境泄漏进交付包

97 个 `.pyc`、多个 `__pycache__`、`egg-info` 被交付。其问题不只是“难看”：

- 暴露本地 Python 版本和测试环境；
- 增大包体、制造重复文件；
- Cursor 可能索引字节码旁的同名源文件；
- 说明打包过程没有 clean-room 检查。

**修复：** 新增 `scripts/build_clean_snapshot.py`，从 Git 索引或白名单生成交付 ZIP；构建后断言不存在 `.pyc`、`__pycache__`、`.egg-info`、`.coverage`、临时报告和密钥文件。

## P1-04：依赖与版本不可复现

`pyproject.toml` 中：

- 版本为 `0.0.0`；
- `pytest` 被列为运行依赖，而不是开发依赖；
- 没有锁文件；
- 没有 README、许可证、作者、分类器等发布元数据；
- 没有 CI 支持矩阵明确 Python 版本。

**影响：** 同一代码在不同日期可能安装到不同依赖版本，测试结果不可严格复现。

**建议：**

- 使用 `[project.optional-dependencies].dev` 或 dependency groups 隔离测试工具；
- 生成锁文件；
- 版本来自 Git tag 或单一版本文件；
- 每份 API 评估归档记录依赖快照。

## P2-01：编码和换行不统一

33 个文本含 BOM、84 个文本含 CRLF。中文路径和中文标识符本身不是错误，但 BOM/CRLF 混用会增加：

- 补丁噪音；
- 哈希不稳定；
- exact_text 证据定位的边界风险；
- Windows 与 Linux 行为差异。

**建议：** `.editorconfig` 固定 UTF-8 无 BOM、LF；仅在确有 Windows 工具要求时例外。

# 六、L1 与语义评估问题

## P0-07：评估脚本存在样本专属分支

`scripts/eval_l1_semantic_golden.py` 明确包含：

```text
CALIBRATION_CHAPTER_IDS = ("GS-001", "GS-005")
```

并针对固定 run-id `L1_Phase2A_RUBRIC_CALIBRATION_20260627` 预先运行这两个样本，失败则阻止全量。

它没有直接写死判定结果，但仍然把具体金标准样本写入通用评估器，与“禁止样本专属硬编码”的治理目标冲突。

**正确设计：**

- 校准目标应来自 CLI：`--chapter-id` 可重复；或来自 manifest 的 `calibration_subset`；
- run-id 只能是标识，不能改变程序语义；
- 通用评估脚本不得知道 GS-001/GS-005。

**置信度：100%。**

## P0-08：“evidence_semantically_sufficient” 名称与能力不符

当前所谓语义充分性主要通过以下规则判断：

- `strength_summary`、`risk_summary`、`final_reason`、`evidence_rationale` 非空；
- `analysis_summary` 包含预设的正面、风险、结论词；
- `final_reason` 命中“信息、对话、场景、重复”等关键词；
- 因果 PASS 必须字面包含“起因、行动、结果”；
- 章末证据在最后 20%；
- rationale 长度达到最低值。

这些只能验证“输出是否按模板写”，不能验证“摘句是否真的支持结论”。模型自己写结论、自己写依据、自己写 rationale，本地程序再检查措辞，属于自证循环。

**具体风险：**

- 一条只证明“少半块”的摘句，可被模型用来宣称完整因果链；
- 只要写出“起因、行动、结果”三个词，即可通过因果充分性形式检查；
- `PHENOMENON_KEYWORDS` 中“信息、对话、场景”等词过于宽泛；
- 合理详细的 `final_reason` 只要包含“情节清晰”也可能被整体拒绝；
- 最后 20% 是位置启发式，不等于章末钩子语义。

**整改方向：**

方案 A（诚实降级）：把字段改名为 `evidence_protocol_sufficient` 或 `evidence_structurally_sufficient`。

方案 B（真正语义判断）：使用独立裁判模型或人工双评，裁判只读取 `dimension + verdict + summary + quote + paragraph context`，判断 `SUPPORTED / PARTIAL / UNSUPPORTED`；不能由原审计模型自评。

**置信度：99%。**

## P0-09：Phase 2A 数据集无法支持强业务结论

当前主集只有 5 章：

- GS-004 是编辑衍生稿，不是未经改写的真实章节；
- GS-001 是真实边界争议；
- GS-005 有完整请求—行动—结果链，更像“成稿质量差”，不是纯粹“叙事不成立”；
- GS-005 同时承担 `clear_fail` 和 `high_lexical_bad_narrative`；
- 人工标签没有独立双盲评审和一致性统计；
- 标签最初由同一审计流程生成，独立性有限。

因此：

```text
Phase 2A 可以测试协议是否工作，
不能证明模型已经校准，
也不能据此估算真实误杀率和漏检率。
```

**建议 Phase 2A.2：**

- v1 完整冻结，不改历史；
- 建 v2，至少 12–15 章用于校准；
- 每类场景至少 2 个样本；
- clear_fail 必须是纯负例；
- 至少两名独立评审者盲标；
- 争议样本记录为 `BORDERLINE`，不强迫单点答案；
- 20–30 章后才进入 Phase 2B 误差率评估。

## P1-07：一次随机采样被当成模型能力结论

L1 默认 temperature 为 0.2。单章各运行一次后，直接输出 `MODEL_CALIBRATION_REJECTED`。

对于生成模型，单次结果不能区分：

- 稳定偏差；
- 随机波动；
- 提示词位置效应；
- 模型别名更新；
- 服务端版本变化。

**建议：**

- 校准评估使用 temperature 0 或明确支持的最确定配置；
- 每个边界样本至少重复 3 次；
- 保存一致率和 verdict 分布；
- 业务验收以“稳定通过”而非单次通过为标准。

## P1-08：评估归档缺少可复现元数据

当前归档有结果和 raw response，但缺少或未稳定记录：

- 请求开始/结束时间和耗时；
- 输入/输出 token；
- API 返回的实际 model 名称；
- request ID；
- 提示词哈希；
- Schema 哈希；
- rubric 哈希；
- 正文哈希；
- 标签哈希；
- Git commit；
- Python 和依赖版本。

没有这些字段，即使同名 run-id 也无法严格复现。

## P1-09：评估归档目录命名失真

`ARCHIVE_ROOT` 固定为：

```text
审计纠偏_2026-06-26/L1_Phase2A_首次真实API评估/
```

但后续 protocol fix、rubric calibration 也写入该目录。“首次真实 API 评估”已经变成所有评估的总目录，名称与内容不符。

建议改为：

```text
运行记录/l1_phase2/
├─ 20260627_first_real_api/
├─ 20260627_protocol_fix/
└─ 20260627_rubric_calibration/
```

## P1-10：`hard_pass` / `hard_fail` 字段语义错误

代码当前：

```python
hard_pass = expected == "PASS" and overall != "FAIL"
hard_fail = expected == "FAIL" and overall != "PASS"
```

这意味着：

- 期望 PASS、实际 REVIEW 或 AUDIT_BLOCKED，`hard_pass` 可能为 true；
- 期望 FAIL、实际 REVIEW 或 AUDIT_BLOCKED，`hard_fail` 可能为 true。

字段名称暗示“实际硬通过/硬失败”，但逻辑只是“没有发生相反极值”。建议删除或改名：

```text
no_catastrophic_false_negative
no_catastrophic_false_positive
```

## P1-11：Phase 2A 没有真正覆盖项目语义上下文

评估脚本调用语义上下文时使用 `project=None`，因此 golden corpus 测试的是单章模型审计，不是：

- manifest 章节顺序；
- prior chapter 上下文；
- 项目 IR；
- 角色状态继承；
- 世界约束继承。

而这些恰恰是小说工程区别于孤立文本分类器的关键价值。

# 七、L2 问题

## P0-10：L2 六个能力模块并不对称

实际结构：

- L2-01：读取正文上下文，调用 DeepSeek，校验证据，输出诊断修复单；
- L2-02～L2-06：基本是调用同一个 `生成标准修复单` 的薄包装，主要依赖预先定义的规则模板。

这并不必然错误，但“六个正式能力模块”的命名让人误以为它们都具备同等语义诊断能力。

**建议重新命名能力等级：**

```text
L2-01 = SEMANTIC_DIAGNOSIS
L2-02~06 = RULE_GUIDED_REPAIR_TEMPLATE
```

然后决定哪些模块确实需要模型诊断，逐个升级，而不是靠文件数量宣称能力完整。

## P1-12：L2 路由存在重叠和措辞依赖

`routes.json` 中：

- “认知成本过高”可命中 L2-02；
- “C高：认知成本过高”可命中 L2-05；
- “字数不足”被路由到 L2-01；
- 路由依赖字符串包含和顺序/长度策略。

**风险：** 同一真实问题只因 failure_type 的措辞不同就进入不同能力模块。

**整改：** failure packet 应先产出稳定的枚举：

```text
NARRATIVE_CAUSALITY
STYLE_READABILITY
CHARACTER_MOTIVATION
WORLD_RULE_CONSISTENCY
MARKET_RETENTION
TECHNICAL_INTEGRITY
```

路由只基于枚举，不基于自然语言关键词。自然语言只用于说明。

## P1-13：L2 内有硬编码特例和死分支

- `能力修复单.py` 对 `L2-05 + 失败类型.startswith("C高")` 特殊设置其他 L2 辅助；
- `L2_99_接口判断.py` 在 NON_L2 分支检查 `target == "回L1.5"`，但当前 NON_L2 映射并不包含该目标，属于结构性死分支或历史残留。

这类代码应由规则数据驱动，不能隐藏在 Python 条件中。

## P1-14：L2-01 证据校验仍是“存在性”，不是根因支持度

L2-01 验证 quote 是否逐字存在，并检查 root cause 不是简单等于输入字段，但并没有验证 root cause 与 quote 的语义支持关系。它与 L1 的问题同源，只是字段更少。

# 八、L3 问题

## P1-15：L3 的“任务规划”与“正文生成”边界矛盾

L3 README 明确说默认入口只生成 `TASK_PLANNING_ONLY` 任务包；但 `候选正文生成.py` 已经调用 DeepSeek 并写出候选正文。

需要显式区分两个运行模式：

```text
PLAN_ONLY
EXECUTE_CANDIDATE
```

不能让“L3 已运行”同时可能表示只规划，也可能已生成正文。

## P1-16：候选正文反作弊能力不足

当前重复检测主要检查完全相同段落/句子的比例；复制检测只在“整个候选正文是正式章节的连续子串”时拒绝。

因此以下情况仍可能通过：

- 同一句式替换少量词后重复几十次；
- 复制正式正文的多个大块并重排；
- 每段改一个字规避 exact duplicate；
- 复制 IR 或其他候选稿；
- 低信息密度扩写。

**建议：**

- 使用 n-gram 重复率、最长公共子序列、段落相似度；
- 检测候选内部自相似和对源文本相似；
- 输出反作弊证据和阈值；
- 高相似不自动判抄袭，但进入人工 REVIEW。

## P1-17：IR 输入截断与选择规则过于粗糙

每个 IR 文件只取前 1500 字，最多前 6 个；正式章节只取前 4000 字。没有：

- 内容优先级；
- 与任务维度的相关性选择；
- 文件哈希；
- 被截断提示；
- token 预算；
- 引用来源映射。

这会导致关键规则恰好位于文件后部时被静默丢弃。

# 九、DeepSeek 客户端与运行可靠性问题

## P1-18：网络错误分类不准确

`urllib.error.URLError` 被统一转换成 `TimeoutError`。DNS 失败、连接拒绝、证书错误等都可能被记录为 TIMEOUT，影响重试策略和诊断。

应区分：

```text
CONNECT_TIMEOUT
READ_TIMEOUT
DNS_ERROR
TLS_ERROR
CONNECTION_REFUSED
HTTP_429
HTTP_5XX
```

## P1-19：缺少调用可观测性

客户端没有稳定记录：

- duration_ms；
- request_id；
- returned_model；
- prompt_tokens / completion_tokens / total_tokens；
- estimated_cost；
- retry_after；
- prompt/schema/input hash。

这正是用户感到“运行时间过长”时无法迅速判断瓶颈的原因。

## P1-20：没有统一的重试、限流和熔断层

L1 自己实现重试，L2/L3 没有统一策略。未来会形成：

- 不同模块对 429、5xx、timeout 的处理不一致；
- 多模块并行时不可控；
- 同一故障被重复调用放大成本。

建议在 DeepSeekClient 上方建立统一 `ModelGateway`，负责重试、指数退避、jitter、限流、熔断、预算和日志。

## P1-21：输出预算未受控

请求没有显式 `max_tokens`/最大输出预算；六维报告又新增多个重复字段。`analysis_summary`、`strength_summary`、`risk_summary`、`final_reason` 存在内容重叠，增加响应时间和费用。

建议删除重复表达：保留 `strength/risk/final_reason`，由本地代码合成展示摘要。

## P2-03：模型别名变化的可复现风险

模型配置依赖环境变量和当前别名。即使名称相同，服务端模型可能升级。归档必须保存 API 返回的真实模型标识和调用日期，不能只保存请求别名。


建议增加 `--allow-external-entrypoint` 明示开关；默认外部项目只能使用共享受信任入口，不能任意执行项目代码。

# 十二、安全、隐私与版权问题

## P1-27：第三方小说全文进入仓库存在版权与泄露风险

- 明确日志保留期限；
- 支持脱敏归档；
- 禁止提交 API Key；
- 将真实正文和模型输出排除在公开交付包外；
- 对日志目录设置 `.gitignore` 与发布白名单。

# 十七、反向压力测试

## 薄弱假设 1：压缩包可能不是用户当前工作树

成立。用户汇报 67 tests，ZIP 实跑为 66 tests，且压缩包没有 Git 元数据。故本报告只对 `XC-UE(12).zip` 负责，不对未打包的本地文件负责。

## 薄弱假设 2：Linux 审计可能夸大或遗漏 Windows 问题

成立。项目主要在 Windows/OneDrive 运行，文件锁、长路径、中文用户目录和 PowerShell 行为无法完整复现。但缺少 project.json、空 wheel、陈旧文档和样本硬编码与操作系统无关，结论不受影响。

## 薄弱假设 3：DeepSeek 实时模型可能已变化

成立。未重新调用 API，模型语义结论依据已有归档。因而本报告不声称当前模型一定仍输出相同 verdict，只判断现有评估设计和归档证据不足。

## 薄弱假设 4：Markdown 可能故意只记录历史阶段

部分成立。如果这些文件明确标为历史归档，陈旧不是错误；但根 README 明确要求 Cursor 首先读取并称其为“当前状态”，因此在现有位置和措辞下仍构成高风险问题。

# 十八、审计结语

XC-UE 当前最大的风险不是“功能太少”，而是“功能、文档、规则、包、测试项目、评估数据都各自进化，缺少一个能够约束它们共同状态的工程核心”。

下一步不应继续扩功能。应先把以下四件事变成唯一且可验证的事实：

```text
1. 项目到底如何被加载；
2. 引擎到底如何被安装和运行；
3. 哪些文件是各层真正的真源；
4. 什么证据才足以证明模型判断正确。
```

这四项完成后，已有 L1/L2/L3 代码才会从“能局部运行的工程集合”转为“可持续迭代的小说生产系统”。
